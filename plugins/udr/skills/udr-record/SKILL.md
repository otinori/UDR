# udr-record — 判断の対話的記録 skill

> **前提:** `.claude/skills/_udr-shared/CONVENTIONS.md` を先読み。未初期化なら先に `/udr-init` を提案。

## 核心

UDR の存在意義は **棄却理由の構造化記録** (BR-007 / D-4)。必ず棄却選択肢を聞き出す。

このスキルには **2 つの動作モード** がある:

| モード | 呼び出し | 用途 |
|---|---|---|
| **通常モード** (default) | `/udr-record` | 決まったことを記録。Phase A→B→C を経て accepted/proposed で保存 |
| **draft モード** | `/udr-record draft` または AI が自律的に判断依頼を記録するとき | まだ人間が決めていない。context + 選択肢候補を記録して `draft` で保存 |
| **完成モード** | `/udr-record --complete <id>` | 既存 draft を開き、decision を追記して昇格させる |

## 文章品質ガイドライン — PJ 外読者テスト

UDR は **「このリポジトリを今日初めて見た経験豊富なエンジニア」が文脈なしで読んでも
判断の背景・選択理由・棄却根拠を理解できる** ことを目的とする。
Claude は各フィールドを下記基準で書き、**下書きを user に提示する前に自己チェック**を実施する。

### 読者テスト（4 問）

Claude は下書き完成後、以下を自問する。ひとつでも "No" なら自分で書き直してから提示する。
user に確認してもらうのは「内容の正確性」であり「わかりやすさ」は Claude が保証する。

1. **context**: プロジェクトを知らない人が「何が問題で、なぜ今か」を理解できるか？
2. **decision**: 結論が最初の 1 文で分かるか？
3. **棄却 rationale**: 「なぜダメか」の根拠（数値・制約・事例）が含まれているか？
4. **verification / 事後追記フィールド**（`verification.criteria` や、後日 `/udr-record --complete` 等で追記する内容）: 記録した検証結果が、**その場限りの一時ファイル・ID・パス**（スクラッチパッド、`/tmp`、対話ログ内だけの略称等）に依存せず、手法と結果だけで単独に理解できるか？ 依存していれば、その参照を捨てて自己完結した文章に書き直す（詳細は下記フィールド別ガイド参照）。

### 読者テストの限界と補強 — 独立コンテキストレビュー（Claude Code）

上記の読者テストは、判断の会話に同席した Claude 自身が自問する形式のため、
**「自分が使った表現が専門用語だと気づけない」自己採点の弱さ**を構造的に持つ
（判断の当事者は自分の略語・前提知識を "当たり前" だと感じやすい）。

Claude Code ではこれを補うため、下書き完成後・user への提示前に
**会話履歴を一切持たない独立コンテキストでのレビュー**を追加する（実行手順は
`## 処理 §4.3 独立コンテキストレビュー` を参照）。会話に同席していないレビュアー
だけが、真の意味で「初見の読者」を模擬できる。

- 対象: Claude Code（Agent tool でサブエージェントを起動できる環境）。
- 非対応環境（Gemini CLI 等サブエージェント機構を持たないツール）では、
  上記の読者テスト（自己チェック）のみをベストエフォートで実施する。
- **役割を複数持たせたレビュー（PM 視点・監査視点等のマルチロールレビュー）は
  本 skill のスコープ外。** 別途 multi-agent レビュー基盤（conclave）との連携を
  将来検討する（本項は単一の「文脈なし読者」視点のみを扱う、UDR-UDR-20260712T0314-47b）。

### フィールド別ガイド

#### `title` — 動詞＋対象＋理由ワードの 3 要素

| ✅ Good | ❌ Bad |
|---|---|
| 「API 認証方式を OAuth 2.0 に統一 — セッション管理の複雑化を回避」 | 「認証方式決定」 |
| 「CI を GitHub Actions に移行 — Jenkins 保守コスト削減のため」 | 「CI 移行」 |

社内固有の略語は初出時に展開する（一般的な略語 API・CI 等は展開不要）。

#### `context` — 以下の観点を 3-6 行で押さえる

1. **何が問題・制約だったか** — 「○○が△△という理由で限界を迎えていた」
2. **なぜ今この判断が必要か** — 「○○リリース前に決める必要があった」「放置すると△△のリスク」
3. **前提知識の補足** — プロジェクト固有の略語・構成は初出時に 1 文で説明

❌ Bad: `既存の設計では問題があったので、新しいアプローチを検討した。`

✅ Good:
```
現行の認証モジュール（2022 年実装、JWT 固定）はマルチテナント対応
（複数顧客が同一インスタンスを共有する構成）が困難であることが判明した。
2026 Q3 の SaaS 版リリースまでに解決しないと、テナント間のデータ混在
リスクが残るため、今サイクルで方針を確定する必要がある。
```

#### `decision` — 結論を先に、理由を後に（倒置ピラミッド）

初行は非技術者でも大意が掴めるよう平易に。技術詳細は 2 行目以降。

❌ Bad:
```
Redis Cluster + Lua スクリプトによる分散ロックを採用し、既存の Postgres
ベースの実装を置き換える。
```
✅ Good:
```
キャッシュ層を Redis に一本化し、高速セッション管理と分散ロックを実現する。
現行の PostgreSQL ベース実装（遅延 50-200ms）を Redis（遅延 < 5ms）に
切り替えることで、将来のスケールアウトにも対応できるようにする。
```

#### `options[].rationale`（採用案）— 比較軸を明示

「なぜこれが最善か」を棄却案との差分で説明する。「○○と比べて△△の点で優る」形式。

❌ Bad: `シンプルで実装しやすいため。`
✅ Good:
```
既存チームの Go スキルセットと合致し、移行コスト（学習・既存コード改修）
が最小になる。代替の Rust 案と比べてパフォーマンスは劣るが、現フェーズの
スループット要件（1,000 req/s）では差が顕在化しない。
```

#### `options[].rationale`（棄却案）— 理由＋根拠のセット必須

単語一語での棄却理由は禁止。「○○が△△という理由（数値・事例・制約）があるため」の形式。

❌ Bad: `コストが高い。`
❌ Bad: `スケールしない。`
✅ Good:
```
月額ライセンス費用が $2,000/月 となり、現フェーズの予算上限（$500/月）を
4 倍超過するため採用不可。要件が拡大した場合は再評価の余地あり。
```
✅ Good:
```
単一ノード構成のため、ユーザー数が 10,000 を超えると垂直スケールの上限に
達することが PoC で確認済み。2027 年の想定負荷（50,000 ユーザー）に耐えられない。
```

#### `verification.criteria` 等の事後追記フィールド — 一時ファイル・ID に依存しない

`verification.criteria` は判断の作成後、実装・検証が進むたびに追記されることが多い。この追記時が
最も §読者テストが漏れやすいタイミング（下書き提示フローの外で更新されるため）。
シミュレーションやテストの結果を書く際、**その場限りのスクラッチパッドファイル・テストデータの
内部 ID・`/tmp` 配下のパス**などをそのまま書いてはいけない。それらは対話終了・セッション終了と
ともに失われ、後から読む人（Claude 自身の将来セッションも含む）には二度と参照できない。
「判断根拠の蒸発を防ぐ」という UDR 自身の存在意義に反するため、**手法（何を対象に何を確認したか）
と結果（定量的な数値・Yes/No）だけで完結する文章**に必ず変換すること。もし裏付けとなる成果物が
リポジトリに永続化されているなら、`code_refs` で実在するパスとして参照してよい。

❌ Bad:
```
表記ゆれパートナー（sim-002/004/006/008/010）を全件発見（recall 5/5）。
```
✅ Good:
```
表記ゆれのある同義語ペア 5 組を含むテストコーパスで検証し、全 5 組で
同義語パートナーを発見できること（再現率 5/5）を確認した。
```

## draft モード — 判断依頼の記録

AI が「ここは人間が決める必要がある」と判断したとき、または user が「まだ決めていないが状況を残したい」と言ったときに使う。

### draft 作成フロー (Phase A')

通常の Phase A の代わりに以下を実施する。§文章品質ガイドラインは同様に適用。

1. **何を決める必要があるか** (`title`) — 「〇〇の選定」「△△方針の決定」形式
2. **なぜ今決める必要があるか** (`context`) — 背景・タイムリミット・放置した場合のリスク
3. **考えられる選択肢** (`options` with `verdict: deferred`) — まだ承認・棄却は決まっていない
   - 各選択肢の pros/cons・rationale（「検討中」旨を明記）を可能な範囲で記録
   - §文章品質ガイドライン: pros/cons は具体的な根拠と数値を盛り込む
4. **AI の推奨案** (任意) — AI として推奨する選択肢と理由を `context` 末尾か採用候補の rationale に添える

**自動設定:**
- `decision`: `「（未定 — 人間判断待ち）」`（placeholder、draft 限定の例外）
- `status`: `draft`
- `authors`: AI のみでも可（BR-002 対象外）
- `/udr-sync` の選択対象から自動除外

**AI が「人間判断が必要です」と伝えるときの定型:**

```
[UDR 判断依頼] {{title}}

{{context の要点}}

選択肢:
  A) {{opt-1.name}} — {{pros 1 点、cons 1 点}}
  B) {{opt-2.name}} — {{pros 1 点、cons 1 点}}

AI 推奨: {{推奨案とその理由（任意）}}

→ draft として記録して後で完成させますか？ (Y) / 今すぐ決めますか？ (N→通常フロー)
```

### draft 完成フロー (`/udr-record --complete <id>`)

1. `.udr/records/<id>.yaml` を Read して内容を提示
2. Phase A 項目 1 を引き継ぎ、`decision` を人間から聞き出す（§文章品質ガイドライン: 結論先）
3. 採用案の `verdict` を `accepted` に、棄却案を `rejected` に更新
4. 棄却 rationale が「検討中」のままなら §文章品質ガイドラインに従い具体化を求める
5. `authors` に人間を追加
6. status 昇格:
   - authors に人間が含まれる → `accepted`（FR-012）
   - AI のみのまま → `proposed`（BR-002 強制）
7. `updated` を今日の UTC に更新して Write
8. 独立コンテキストレビュー（§4.3 と同じ手順）を実施
9. audit.log に `op: "update"`, `changed_fields: ["decision","options","status","authors","updated"]` を追記
10. 通常の同期トリガー（Step 7）を実行

## 対話フロー (FR-012)

### Phase A: 必須 3 項目 (これだけで最小 UDR が完成)

user から以下を順に聞き出す。既に会話文脈で明らかな項目は再質問せず、確認だけ取る。

1. **何を決めたか** (`title` + `decision`)
   - title は 50 字以内の一行（§文章品質ガイドライン: 動詞＋対象＋理由ワード）。decision は 2-3 行（結論先）。
2. **なぜ決めたか** (`context` + 採用案の `rationale`)
   - context = 判断に至った背景（3-6 行）。§文章品質ガイドライン: 問題・なぜ今・前提補足の 3 点を押さえる。
3. **他にどんな選択肢があり、なぜそれを棄却したか** (`options[].verdict: rejected` + `rationale`)
   - §文章品質ガイドライン: 棄却理由は「理由＋根拠（数値・制約・事例）」のセットで書く。単語一語禁止。
   - **棄却が 0 件**なら「本当に選択の余地がなかったか？」を 1 回だけ確認。
   - それでも 0 件なら警告付きで進む（FR-015、ブロックはしない）。

4. **この判断を縛る外部制約はあるか？** (`constraints`) ← Phase A で必ず確認
   - 例: コスト上限・技術スタック制限・法規・既存方針・リリース期限。
   - 「なし」と明確に確認してから次へ進む（後から読む人にとって最重要コンテキスト）。
   - ある場合はダブルクォートで記録: `- "予算上限: $500/月 (2026 Q3 まで)"`

5. **関連する既存判断はあるか？** (`relations` 軽量確認) ← Phase A で必ず確認
   - `/udr-search <キーワード>` で 30 秒確認するよう案内。
   - ID が出たら `depends_on` / `triggers` を記録（`relationship` に関係の性質も添える）。
   - 「わからない / なし」なら Phase B 項目 6 で改めて深堀りする。

### Phase B: 任意 6 項目 (user が "skip" と言えば飛ばす)

各項目ごとに「記録しますか？ (skip可)」と聞く。**skip された項目は YAML から キー自体を省略** (PO2-02)。

6. **影響度** (`severity`: high/medium/low) + `consequences` (positive/negative/risks)
7. **関連判断の深堀り** (`relations.depends_on` / `relations.triggers` — Phase A 項目 5 で未確定分)
   - user が ID を挙げたら **循環参照チェック** (FR-007)。
   - `relationship` フィールドに関係の性質を記録: 「技術的依存」「ビジネス要件から」「方針遵守」等。
8. **追加の選択肢** (`options` の追記)
9. **コード参照** (`code_refs`) — この判断に関係するファイル・実装箇所
   - 例: `src/auth/TokenValidator.java` / `migrations/0042_user_schema.sql`
   - skip した場合、`domain: architecture / design` では `/udr-review` で後から記入を促される。
10. **タグ** (`tags`) — スプリント番号・マイルストーン等の自由タグ
    - 例: `sprint-7`, `milestone-v2.0`, `tech-debt`
11. **最終判断者** (`owner`) — authors の中で最終責任者を明示する場合

### Phase C: メタ確定

- `domain` を enum (`architecture/requirements/design/risk/project/operations/other`) から選ばせる。会話文脈から推定値を提示し、承認ベースで決める。
- `authors` を確認。現セッションの user 名 (`git config user.name` で取得) + AI (`claude-code` 等) を自動追加。
- `pinned` は明示的に user が "pin" と言ったときのみ true。既に 5 件 pinned なら警告。
- **`claude_summary` 執筆の促し:**
  「AI がコンテキストファイル (CLAUDE.md 等) で読む要約を 2-3 行で書きますか？
   書かない場合はテンプレート自動生成になりますが、判断のニュアンスが失われることがあります。」
  → 書いた場合: `claude_summary` フィールドに格納（sync 時に優先使用）。
  → skip: `null` のまま（従来動作）。

## 処理

### 1. ID 採番 (FR-001 / AD-05)

```
UDR-<config.repo_short>-<UTC YYYYMMDDTHHmm>-<rand3 [0-9a-f]>
```

衝突時は rand3 を 3 回までリトライ。3 回失敗ならエラー `UDR-REC-001`。

### 2. status 決定 (BR-002 / BR-004)

```
draft モード? ──Yes──▶ status: draft (BR-002 対象外)
     │
     No
     ▼
authors に人間が含まれる? ──No──▶ status: proposed (BR-002強制)
             │
            Yes
             ▼
FR-012 フロー経由 ──▶ status: accepted
```

Phase 1 PoC ではこの skill 経由 = FR-012 扱い。
draft 完成フロー (`--complete`) は通常フローと同じ status 決定を再適用する。

### 3. summary_hint 自動生成 (FR-009 Step 1, C-007)

テンプレート結合のみ（LLM 呼び出し禁止）:

```
[<id>] <domain[:3]>: <title>。
決定: <decision の最初の文>。
棄却: <rejected[0].name>(<rejected[0].rationale の 20 字以内>)<, rejected[1].name ...>。
制約: <constraints[0] の 20 字以内>。 (constraints がある場合のみ追加)
```

`claude_summary` は Phase C で user が記述した場合のみ設定（sync 時の優先度最高）。`client_summary` は null。

### 4. YAML 書き出し

- パス: `.udr/records/<id>.yaml`
- スキーマは `CONVENTIONS.md §3` に厳密準拠
- skip された項目は **キー省略** (PO2-02)
- `date` / `updated` は UTC ISO 日付 (`YYYY-MM-DD`)

#### 4.1 スカラー出力規約（**必須・パース破壊防止**）

生成した YAML は標準パーサ (PyYAML `safe_load` 等) で**必ず**読めること。プレーンスカラーに `:␣`（半角コロン+空白）や特殊文字が混入すると構文エラーになるため、以下を厳守する:

- **フリーテキストの複数行**（`context` / `decision` / `options[].rationale` / `consequences.*` の長文など）は **ブロックリテラル `|`** で書く。
- **リスト項目**（`constraints` / `options[].pros` / `options[].cons` / `consequences.{positive,negative,risks}` 等の `- …`）は **必ずダブルクォート**して `- "…"` 形式にする。
- **単一行スカラー**（`title` / `options[].name` など）は、次のいずれかを含むなら**必ずダブルクォート**: 半角 `:` の後に空白 / `#␣` / 先頭が `- ? : , [ ] { } & * ! | > ' " % @ \`` のいずれか / 先頭・末尾の空白 / `: ` `[` `]` `{` `}` を本文に含む。**簡便則: 値に `:␣` を含むなら無条件でクォート**。
- ダブルクォート内では `"` を `\"`、`\` を `\\` にエスケープする。
- 迷ったらクォートする（過剰クォートは無害、未クォートは破壊的）。日本語の全角コロン `：` は安全だが、半角 `:` 混入に注意。
- **書き出し後の自己検証**: 生成ファイルを 1 度パースできるか確認し（不可なら該当スカラーをクォートして再出力）、その上で user に提示する。

> 例: `- "(現状: 自動投入 → 人手 Enter で確定)"` ✅ / `- (現状: 自動投入 …)` ❌（`: ` で mapping と誤認され破綻）

### 4.2 supersedes 指定時の旧レコード backlink 書き込み

`supersedes: [旧ID, ...]` が記録された場合、各旧 ID に対して:

1. `.udr/records/<旧ID>.yaml` を Read（不在なら警告のみ、スキップ）
2. `relations.superseded_by` リストに新 ID を追加（既にあればスキップ）
3. `updated` を今日の UTC 日付に更新
4. Write で上書き（他フィールドは変更しない）
5. `audit.log` に追記:
   ```json
   {"ts":"<UTC>","actor":"<name>","role":"<role>","op":"supersede","id":"<旧ID>",
    "changed_fields":["relations.superseded_by","updated"]}
   ```

これにより旧レコードから後継へ直接辿れる双方向リンクが確立される。

### 4.3 独立コンテキストレビュー（フレッシュレビュー、Claude Code）

YAML 書き出し（4.1 の自己パース検証まで完了）した直後、user へ提示する前に実施する。
背景は §文章品質ガイドライン「読者テストの限界と補強」を参照。

1. **Agent tool でサブエージェントを 1 体起動する。** 渡すコンテキストは
   **書き出した YAML ファイルの内容のみ**。この会話の履歴・背景説明・意図の
   要約などは一切渡さない（渡すと "文脈なし読者" のシミュレーションが崩れる）。
2. サブエージェントへの指示文（そのまま渡す）:
   ```
   あなたはこのリポジトリを今日はじめて見た第三者エンジニアです。
   この会話の背景は一切知りません。以下の UDR ドラフトだけを読んで判定してください。

   (a) context だけで「何が問題で、なぜ今この判断が必要か」が分かるか
   (b) decision の結論が最初の1文で分かるか
   (c) 棄却された選択肢それぞれに、具体的根拠（数値・制約・事例）があるか
       （「コストが高い」のような単語一語の理由は不可）
   (d) 説明なしの略語・社内固有名詞・一時的なファイルパスやID（/tmp 配下、
       スクラッチパッド等）への依存がないか

   ひとつでも不明瞭な箇所があれば、フィールド名と該当箇所を指摘してください。
   問題なければ「指摘なし」とだけ返してください。
   ```
3. 指摘があれば、その指摘を踏まえて Claude 自身が Edit で YAML を修正し、
   **もう一度だけ**同じレビューを新しいサブエージェントで実行する（無限ループ
   防止のため最大 2 回）。
4. 2 回目も指摘が残る場合はブロックせず、指摘内容を保持したまま次の user 提示
   ステップで「未解消のレビュー指摘」として併せて提示する（FR-015 と同じ
   "警告のみで進行" の方針）。
5. サブエージェント機構を持たない環境（Gemini CLI 等）では本ステップを丸ごと
   スキップし、§文章品質ガイドラインの自己チェックのみとする。

`--complete` フロー（draft 完成）でも、Write 後・同期トリガー前の同じ位置で実施する。

### 5. DAG 検証 (FR-007)

`depends_on` / `triggers` が指定されていたら:

1. 参照先 ID の実在確認（`.udr/records/<id>.yaml` を Glob）
   - 不在なら警告のみ（ブロックしない、FR-001 受入基準）
2. 循環検出
   - 新 UDR を仮想ノードとして既存 DAG に追加し、DFS で循環を探索
   - 循環あり → エラー `UDR-REC-007` + 循環パス提示してブロック

### 6. audit.log 追記

```json
{"ts":"<UTC ISO>","actor":"<name>","role":"human","op":"create","id":"<id>","changed_fields":["*"]}
```

### 7. 同期トリガー (FR-012 受入基準 (3))

記録完了後、自動で `/udr-sync` を呼ぶ提案を user に提示（"このまま sync しますか？" Y/N）。

## user への確認事項チェックリスト

- [ ] Phase A 必須 3 項目のヒアリング完了
- [ ] 棄却選択肢 1 件以上、または "選択の余地なし" 確認済み
- [ ] constraints: 外部制約の有無を確認済み（「なし」の場合も確認）
- [ ] relations: Phase A 項目 5 で関連既存判断を確認済み
- [ ] domain 承認済み
- [ ] authors 承認済み
- [ ] claude_summary: 執筆 or skip を確認済み
- [ ] Phase B 6 項目は skip/記録を明示確認
- [ ] supersedes 指定時: 旧レコードへの superseded_by 書き込み完了
- [ ] 独立コンテキストレビュー（§4.3）実施済み（Claude Code）／非対応環境では該当なし
- [ ] 生成した YAML を user に提示し承認取得（読者テスト 4 問を自己チェック済み、未解消のレビュー指摘があれば併記）
- [ ] 書き出し後、ファイルパスと ID を伝達

## エラーコード

| Code | 条件 |
|---|---|
| UDR-REC-001 | ID 衝突がリトライ上限を超過 |
| UDR-REC-002 | `.udr/` 未初期化 → `/udr-init` を案内 |
| UDR-REC-003 | domain が enum 外 |
| UDR-REC-004 | status 遷移違反（新規では通常発生しない） |
| UDR-REC-005 | pinned が既に 5 件 |
| UDR-REC-007 | DAG 循環検出 |
| UDR-REC-015 | rejected 0 件 (警告のみ、記録は続行) |
| UDR-REC-020 | `--complete` 対象 ID が draft でない (proposed 以上は通常の update を使うこと) |
| UDR-REC-021 | draft の `decision` フィールドが placeholder のまま完成フローを呼んだ (自動検出) |
